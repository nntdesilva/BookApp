# Version 0.11.5 - 2026-03-11

## Bug Fix: Diagnostic Logging for Login Redirect Loop

Added targeted diagnostic logging to the gateway auth middleware and login/signup routes to pinpoint why a successful login (status=200, success=true from the auth-service) was silently redirecting users back to `/login` instead of landing on `/`.

## What's New

### `gateway/middleware/auth.js`

- **`verifyToken`** — logs whether the `token` cookie is present in the incoming request and all other cookie names found; if `jwt.verify` fails, logs the error name, message, and the local secret length so a secret mismatch between services is immediately visible in CloudWatch
- **`requireAuth`** — logs the method and path of the request that was rejected and redirected to `/login`
- **`redirectIfAuth`** — logs when an already-authenticated user is redirected away from the login page

### `gateway/app.js`

- **`POST /login`** — logs the cookie name, token length, and cookie options (`secure`, `sameSite`, `httpOnly`) at the moment the cookie is set, confirming the response carries the correct `Set-Cookie` header
- **`POST /signup`** — same cookie-set log as login
- **`GET /`** — a pre-`requireAuth` middleware step logs exactly which cookie names the browser sent on the redirect, revealing whether the `token` cookie arrived at all

## Files Changed

- **`gateway/middleware/auth.js`** — added log lines to `verifyToken`, `requireAuth`, and `redirectIfAuth`
- **`gateway/app.js`** — added cookie-set log after `res.cookie()` in `POST /login` and `POST /signup`; added cookie-presence log middleware on `GET /`
