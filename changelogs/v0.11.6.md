# Version 0.11.6 - 2026-03-12

## Chore: Structured Logging with Pino Across All Services

Replaced all ad-hoc `console.log/warn/error` calls with [pino](https://getpino.io/) structured JSON logging across every service. Every log entry is now a machine-readable JSON object with a consistent shape, making it filterable and queryable in CloudWatch Logs Insights and any other log aggregator.

## What's New

### `config/logger.js` (new file in every service)

Each service now has its own pino logger configured with:

- `base: { service: "<service-name>" }` — the service name is embedded in every log entry automatically
- `pino-pretty` transport when `NODE_ENV !== "production"` — colorized, human-readable output for local development with timestamps
- Plain newline-delimited JSON in production — one JSON object per line, exactly what CloudWatch expects

### Log shape

Every log entry now looks like this in production:

```json
{"level":"info","time":1741788000000,"service":"auth-service","component":"authRoutes","event":"login_success","username":"bob","userId":"abc123","tokenLength":245}
```

And in development (via pino-pretty):

```
12:34:56.789 INFO  login_success  username=bob userId=abc123 tokenLength=245
```

### Structured fields vs. string interpolation

All string template literals like `[service] event — key=val key2=val2` have been converted to structured objects: `logger.info({ event: "...", key: val, key2: val2 })`. This means every field is individually queryable — no regex parsing required to extract a value from a log line.

### Error serialization

`console.error("msg:", error.message, error.stack)` calls have been replaced with `logger.error({ event: "...", err: error })`. Pino has a built-in error serializer that expands the `err` field into `{ type, message, stack }` automatically.

### Child loggers per module

Each file creates a child logger with its module name:

```js
const logger = require("../config/logger").child({ component: "authRoutes" });
```

This adds `component` to every log line from that file, so you can filter by both `service` and `component` to zero in on exactly the module that failed.
