# Version 0.10.1 - 2026-02-26

## MCP for Claude Code + AWS Deployment

The MCP server now supports Streamable HTTP transport alongside stdio, enabling remote deployment for Claude Code (and any HTTP-based MCP client). Deployed to AWS App Runner with a dedicated CI/CD workflow that only redeploys when MCP-related files change.

## What's New

### Streamable HTTP Transport (`mcp-server.js`)

The MCP server now supports two transport modes controlled by the `MCP_TRANSPORT` environment variable:

| Mode | Use Case | Client |
|---|---|---|
| `stdio` (default) | Local development | Cursor |
| `http` | Remote deployment | Claude Code, any HTTP MCP client |

In HTTP mode, the server runs an Express app with:

- `GET /health` — health check endpoint for App Runner
- `ALL /mcp` — Streamable HTTP transport endpoint with session management
- `MCP_API_KEY` bearer token authentication
- Automatic session creation, tracking, and cleanup

### Docker & AWS Deployment

- **`Dockerfile.mcp`** — multi-stage Alpine build that runs the MCP server in HTTP mode on port 8080
- **AWS App Runner** — deployed as a separate App Runner service (`book-app-mcp`) alongside the main app
- **ECR Repository** — images pushed to a dedicated `book-app-mcp` ECR repository

The MCP server runs as a separate App Runner service from the main web app, with its own:

- ECR repository (`book-app-mcp`)
- App Runner service ARN (`MCP_APPRUNNER_SERVICE_ARN`)
- ECR access role (`ECR_ACCESS_ROLE_ARN`)
- API key (`MCP_API_KEY`) for bearer token auth

### MCP Deployment Workflow (`.github/workflows/deploy-mcp.yml`)

A dedicated GitHub Actions workflow handles the full MCP deployment pipeline:

1. **Tests** — unit and integration tests run against a MongoDB service container
2. **Build** — Docker image built from `Dockerfile.mcp` and pushed to Amazon ECR (`book-app-mcp`)
3. **Deploy** — App Runner service updated via `aws apprunner update-service`
4. **Health check** — `GET /health` passes on the live service

The workflow uses path-based triggers — the MCP server only redeploys when one of its dependency files changes:

| File | Reason |
|---|---|
| `mcp-server.js` | Server entry point |
| `Dockerfile.mcp` | Docker build definition |
| `services/favoriteService.js` | Favourites CRUD |
| `services/gutenbergService.js` | Gutenberg word search |
| `services/analysisService.js` | Code execution analysis |
| `services/embeddingService.js` | Semantic embeddings (transitive) |
| `config/database.js` | MongoDB connection |
| `config/appConfig.js` | App configuration |
| `models/User.js` | User model |
| `package.json` / `package-lock.json` | Dependencies |

Unrelated changes (CSS, templates, controllers, views, etc.) no longer trigger an MCP redeploy.

### Claude Code Configuration (`.mcp.json`)

A root-level `.mcp.json` configures the remote MCP server for Claude Code:

| Before (v0.10.0) | After (v0.10.1) |
|---|---|
| `stdio` transport, local `node mcp-server.js` | Streamable HTTP via AWS App Runner |
| `BOOK_APP_USERNAME` env var passed locally | Bearer token authentication |
| Only works on the dev machine | Accessible from any Claude Code session |

Claude Code now connects to the MCP server remotely — no local Node.js process required.

## Files Changed

- **`mcp-server.js`** — added `startHttp()` function with Express server, session management, and auth middleware
- **`Dockerfile.mcp`** — new file; multi-stage Docker build for MCP HTTP server
- **`.github/workflows/deploy-mcp.yml`** — new file; standalone deployment workflow with path filters
- **`.github/workflows/ci.yml`** — removed `deploy-mcp` job (moved to dedicated workflow)
- **`.mcp.json`** — updated from local stdio config to live AWS App Runner endpoint with bearer auth
- **`package.json`** — added `start:mcp` script

All nine MCP tools (favourites CRUD, Gutenberg word/related-word search, book statistics, and Plotly visualizations) are live and accessible remotely. The stdio transport for Cursor is fully unchanged.
